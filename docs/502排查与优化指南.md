# 502 Bad Gateway 排查与优化指南

适用场景：站点运行数天后偶发或频繁出现 `502 Bad Gateway`。

## 一、先明确 502 的本质

`502` 一般不是 OpenResty 本身故障，而是 OpenResty 作为网关从 PHP-FPM 没拿到正常响应：

1. PHP-FPM 进程挂掉/不足
2. PHP 执行过慢导致 upstream 超时
3. 外部接口调用卡死，长期占用 FPM worker
4. MySQL 慢查询把请求拖慢

## 二、现场排查命令（优先级从高到低）

### 1) OpenResty/Nginx 错误日志

```bash
tail -f /www/wwwlogs/*.error.log /www/server/nginx/logs/error.log /usr/local/openresty/nginx/logs/error.log 2>/dev/null
```

重点关键词：

- `connect() to unix:/...php-fpm.sock failed`
- `upstream timed out`
- `recv() failed (104: Connection reset by peer)`

### 2) PHP-FPM 日志

```bash
tail -f /www/server/php/*/var/log/php-fpm.log /var/log/php*-fpm.log 2>/dev/null
```

### 3) 进程与连接状态

```bash
ps -ef | egrep 'php-fpm|nginx|openresty' | grep -v grep
ss -lntp | egrep 'php-fpm|nginx'
top -H -p $(pgrep -d, php-fpm)
```

### 4) MySQL 慢查询日志

```bash
tail -f /www/server/data/*.slow.log /var/log/mysql/*slow*.log 2>/dev/null
```

## 三、项目内已做的优化点

### 1) 定时任务查询改为可走索引

- 文件：`cron.php`
- 调整：
  - `notify/notify2` 任务不再对 `endtime` 使用函数包裹条件（减少全表扫描风险）
  - 增加 `ORDER BY`，优先处理到期重试记录
  - 修复失败统计查询缺少 `notify` 字段的问题

### 2) 全局 HTTP 请求增加超时保护

- 文件：`includes/functions.php`
- 调整：
  - `CURLOPT_CONNECTTIMEOUT=10`
  - `CURLOPT_TIMEOUT=30`

作用：避免上游接口长时间不响应时占满 PHP-FPM worker。

### 3) 补充核心索引（pre_order）

- 文件：`install/install.sql`
- 文件：`install/update3.sql`
- 新增索引：
  - `status_addtime(status, addtime)`
  - `notify_notifytime_endtime(notify, notifytime, endtime)`
  - `uid_status_date(uid, status, date)`
  - `channel_status_date(channel, status, date)`
  - `buyer_date_status(buyer, date, status)`

## 四、已上线站点的手动索引升级 SQL

如果你的网站数据库已是最新版本（不会再走 `install/update.php`），请手动执行：

```sql
ALTER TABLE pre_order ADD INDEX status_addtime (status, addtime);
ALTER TABLE pre_order ADD INDEX notify_notifytime_endtime (notify, notifytime, endtime);
ALTER TABLE pre_order ADD INDEX uid_status_date (uid, status, date);
ALTER TABLE pre_order ADD INDEX channel_status_date (channel, status, date);
ALTER TABLE pre_order ADD INDEX buyer_date_status (buyer, date, status);
```

注意：如果提示索引已存在，可忽略该条。

## 五、推荐参数模板（宝塔常见场景）

以下是通用建议，需按服务器内存与并发调优：

### 1) OpenResty/Nginx

```nginx
proxy_connect_timeout 5s;
proxy_send_timeout 30s;
proxy_read_timeout 30s;
fastcgi_connect_timeout 5s;
fastcgi_send_timeout 30s;
fastcgi_read_timeout 30s;
```

### 2) PHP-FPM（www.conf）

```ini
pm = dynamic
pm.max_children = 50
pm.start_servers = 8
pm.min_spare_servers = 8
pm.max_spare_servers = 16
pm.max_requests = 500
request_terminate_timeout = 60s
request_slowlog_timeout = 5s
slowlog = /tmp/php-fpm-slow.log
```

### 3) MySQL

```ini
slow_query_log = 1
long_query_time = 1
```

## 六、上线步骤建议

1. 先执行数据库索引 SQL
2. 部署代码变更（`cron.php`、`includes/functions.php`）
3. 重载 PHP-FPM 和 OpenResty
4. 观察 24 小时错误日志与慢日志

## 七、回滚方案

如需快速回滚：

1. 回退 `cron.php` 与 `includes/functions.php` 到改动前版本
2. 索引可保留（一般不会引发功能回退问题）
